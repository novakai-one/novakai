Testing types applicable to novakai
#	Testing type	Impact	Effort	Auto / Manual	Problem it solves	What it does / how it works (novakai-specific)
1	Static type checking (tsc --noEmit) ✅ have it	High	Low	Automated	Catches type errors, null-derefs, bad imports before runtime.	You already run this via npm run typecheck. Compiler walks src/ with strict: true, noUnusedLocals, etc. Cheapest safety net you have.
2	Unit testing (pure-logic modules)	High	Low	Automated	Verifies isolated functions/classes behave correctly without DOM or WASM. Fast feedback loop.	Use Vitest. Test pure functions in io/mermaid.ts (parse→AST→string), io/layout.ts (layout math), io/export.ts (serialization), render/avoidRouter.ts (port-edge math, minus the WASM call). Example: feed a .mmd string, assert the parsed node/edge graph equals expected. Runs in ms.
3	Integration testing (DOM-bound modules)	High	Med	Automated	Verifies modules that touch the DOM/canvas/events actually work end-to-end inside a simulated browser, without spinning up a real browser.	Vitest + happy-dom (or jsdom). Mount interaction/pointer.ts, keyboard.ts, clipboard.ts, panel/inspector.ts against a real HTMLElement tree, synthesize pointerdown/keydown/copy events, assert state changed. Catches the bugs unit tests can't (event wiring, DOM mutations).
4	Property-based testing	High	Med	Automated	Catches edge cases you'd never think to write a test for — round-trips must hold for all valid inputs, not just the 5 examples you hand-picked.	Use fast-check with Vitest. Killer use case for Novakai: Mermaid round-trip property — for any valid novakai doc D, serialize(parse(D)) === D (after normalization). Also: "any node graph renders without throwing," "layout never produces overlapping bounding boxes." Generates hundreds of random inputs per run.
5	End-to-end (E2E) testing	High	High	Automated	Verifies the whole app — bundle, WASM load, canvas render, interactions — works as a user experiences it, in a real browser.	Playwright. Script: launch vite preview, load index.html, drag a node from A to B (Playwright can synthesize drag via mouse moves), wire two nodes, switch theme, export .mmd, assert the exported string contains expected nodes. Catches integration regressions unit/integration tests miss (Vite bundling, WASM loader, CSS layout).
6	Visual regression testing	Med	Med	Automated	Catches unintended visual changes — a wire reroutes weirdly, a panel shifts 4px, a theme color drifts — that functional tests are blind to.	Playwright screenshot assertions (expect(page).toHaveScreenshot()). Snapshot the rendered canvas/panels for: default theme, dark theme, a crowded graph, a selected node. On each run, diff against the stored baseline; fail if pixels differ beyond a threshold. Store baselines in git.
7	Performance / benchmark testing	Med	Med	Automated	Catches regressions in the expensive paths — WASM route-avoidance, layout, re-render on every pointer move — before users feel jank.	Vitest bench (or tinybench). Benchmark render/avoidRouter.ts on a 50-node / 200-wire graph; benchmark io/layout.ts on a 100-node graph. Track p99 ms. Run in CI; fail if a PR slows routing by >15%. Especially valuable because avoidRouter calls into WASM — perf is non-obvious.
8	Contract / spec drift testing ✅ have it	Med	Low	Automated	Ensures the architecture map (.mmd) and the code stay in sync — no orphan modules, no unplanned code, no silent drift.	You already run spec:gate and novakai:verify. This is a form of architectural testing. Consider adding it to CI on every PR so drift fails the build.
9	Accessibility (a11y) testing	Med	Low	Automated (with manual follow-up)	Ensures the diagram editor is usable with keyboard / screen readers, and meets WCAG where applicable.	axe-core injected via Playwright (AxeBuilder). Also a pure keyboard test in Playwright: tab through panels, open context menu, delete a node — all without mouse. Novakai's keyboard.ts makes this especially relevant.
10	Mutation testing	Low–Med	High	Automated	Measures how good your test suite actually is — if tests still pass when you break the code, the tests are weak.	Stryker. Mutates io/mermaid.ts (flips === to !==, removes statements), reruns your unit tests. Mutation score = % of mutations caught. Use after you have a real unit suite; it's a suite-quality tool, not a starter tool.
11	Smoke testing	Med	Low	Automated	Fast "is the build broken?" check that runs on every commit/PR — a tiny subset of E2E.	A single Playwright test: load app, assert canvas rendered, assert no console errors, assert one node visible. Runs in ~3s. Gate PRs on it. Subset of #5, not a separate framework.
12	Manual exploratory testing	Med	Low (per session)	Manual	Catches the things no automated test can anticipate: "this feels wrong," weird edge interactions, real user-flow friction.	You (or a tester) use the app like a user: build a real flowchart, stress-test undo/redo, try broken Mermaid imports, drag fast, multi-select, paste. Keep a short checklist. No framework needed — just disciplined note-taking.
